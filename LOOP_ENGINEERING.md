# Loop Engineering 设计准则（给作者）

> 这是一份"方法论 README"，不是规则本身。硬规则在 `.trae/rules/nanoquant_loop.md`（项目规则，AI 始终生效）。本文件回答：**Loop Engineering 是什么、为什么要这么组织、以后怎么给这个项目设计新 loop**。
>
> 理论来源：Addy Osmani / Boris Cherny (Anthropic) / Peter Steinberger 2026-06 提出；五移动+六部件+生成器/评估器分离的框架来自 Anthropic 工程实践。

---

## 1. 一句话定义

> **Loop Engineering = 设计那个"自动给 agent 喂 prompt"的系统，而不是自己一行行喂 prompt。**
> 你不再是 loop 里的那个"按回车的人"，你是设计 loop 的人。

你以前半夜起来看 Claude Code 跑偏、调一下 prompt 再睡——恭喜，你已经是 loop 了，只不过**你自己就是那个 loop**。Loop engineering 就是把"你"从这个位置上撤下来。

---

## 2. 四层栈（每层管的事不同，失败方式不同）

| 层 | 管什么 | 核心问题 | 失败 blast radius |
|---|---|---|---|
| Prompt eng. | 写好一句话 | 该告诉模型什么 | 一次答错，当场改 |
| Context eng. | 窗口里放什么 | 检索/摘要/清理什么 | 答案自信地错，清上下文 |
| Harness eng. | 武装一次运行 | 给哪些工具、什么算完成 | 一次 run 改了文件，人审 diff |
| **Loop eng.** | **让它在 harness 上自己一轮轮跑** | **怎么让它自己跑起来** | **错被写进状态文件，下一轮当事实，几天才发现** |

**最关键的直觉**：错误成本随它存活的轮数指数增长。Loop 是最大化轮数的机器。所以 loop 层的工程量，**绝大部分不是"怎么让它跑"，而是"在它里面装一个能说不的东西"**。

> 我们这个项目目前主要在前三层。Loop 层的"自动调度"还没真正建（调度层还是裸脚本 + 人按回车）。但我们已有的"探查→补数→EDA→mock→回补→EDA"这套手艺，**结构上就是 loop，只是触发器是人**。本文件就是把这套隐性 loop 显性化、规则化。

---

## 3. 一个 loop 转一圈的五个移动

每一圈做五件具体的事，**少任何一个 loop 都转不起来或原地空转**：

| 移动 | 做什么 | nanoquant 里的对应 |
|---|---|---|
| **发现 Discovery** | 自己找到这轮该干什么（不靠人喂清单） | MCP 探查 tushare 字段；读 panel schema 找缺口 |
| **移交 Handoff** | 把任务交给干活的 agent，隔离干净 | 写一个 Calculator；mock DataFrame 先行；一次一个模块 |
| **验证 Verification** | 换一个视角看结果对不对（能说不的东西） | EDA、行数、主键重复、穿越检查、真实成本回测 |
| **持久 Persistence** | 结果落到对话外（表/文件/水位） | etl_biz_date 水位、overwrite 落库、signal_rebalance |
| **调度 Scheduling** | 让它一轮轮转起来 | sync.py 增量、run_compute 拓扑、周频调仓 |

> **验证是最容易被偷工、也最省不起的一步。** 没有真检查的 loop，就是"agent 给自己点赞"。

---

## 4. 生成器/评估器分离（最难也最值钱的一条）

> 实证：让 agent 给自己的输出打分，它会表扬自己。调一个**独立的、怀疑的评估器**，远比让生成器学会自我批评可行。

在我们的语境里翻译成可操作条款：

- 写代码（生成器）和"判断对不对"（评估器）必须是**两个动作**，最好两套检查。
- "我跑通了"不算验证——必须给出**独立可复查的证据**：行数、null 统计、主键重复数、穿越检查、值域。
- 主键无重复必须**数据论证**（`duplicated().sum()`），不是"应该没重复吧"。
- 防穿越必须**显式验证**（`f_ann_date <= snapshot_date`、`merge_asof direction='backward'`），不是"应该没穿越吧"。
- 回测信号在真实成本引擎建好前，**一个都不能信**（向量化回测虚高 2-5 点/年）。

---

## 5. 设计新 loop 的配方（动手前自检这 5 条）

1. **触发器**：什么启动它？定时 / 事件 / 手动。没有触发器 = 你还在 loop 里。
2. **可验证目标**：怎么判断"完成"？行数 / 测试绿 / 穿越检查 / 夏普。**没有可验证目标 = 你造的是 token 焚烧炉，不是 loop。**
3. **能说"不"的检查**：loop 内部装了什么**独立**检查能停掉它？没有 = agent 自己点头。
4. **状态持久**：结果写到对话外（表 / 文件 / 水位），下次能接着跑。状态只在上下文窗口里 = 不是 loop，是一次性任务跑很多次。
5. **blast radius**：错了能回滚吗？错误能存活几轮才被发现？（越久越致命）

> 经验法则：**先做一个跑一次就停的版本**，把五个移动都跑通，再加定时器让它自己转。一上来就上自动调度 = 把没验证的错自动化。

## 6. 四个会悄悄累积的成本（定期自检）

| 成本 | 表现 | 我们的对策 |
|---|---|---|
| **验证债** | 检查越攒越松，"差不多就行" | 每个验证门是硬规则（项目规则 §2-4），不可省 |
| **理解腐烂** | 代码越跑越久，没人记得它为什么这么写 | 指标列必须注释物理意义 + 公式来源；踩坑进 `DEV_GUIDE.md §7` |
| **认知投降** | 习惯了 loop 自己跑，不再质疑输出 | "生成器≠评估器"；作者可随时要求"给我证据" |
| **token 爆炸** | loop 无脑转，烧钱 | 个人项目暂不自动调度；先跑通手动 loop 再说 |

## 7. nanoquant 现状映射（哪些 loop 已有，哪些缺）

| Loop | 触发器 | 验证门 | 持久 | 调度 | 状态 |
|---|---|---|---|---|---|
| Ingest 接入层 | 人按回车（sync.py） | MCP 探查 + EDA + 主键论证 + 防穿越 ✅ | 水位 + overwrite ✅ | 增量+逐年回补 ✅ | **已有，是隐性 loop** |
| Compute 加工层 | 人按回手（run_compute） | mock + 单日→月→年 + EDA ✅ | overwrite + output_schema ✅ | 拓扑排序 ✅ | **已有，是隐性 loop** |
| Strategy 策略层 | 人按回车 | 真实成本回测 ❌（向量化虚高） | signal_rebalance ✅ | 周频 ✅ | **验证门缺最大**（见 ROADMAP 1.1） |
| 因子检验 loop | — | IC/ICIR/分层/衰减 ❌ | — | — | **未建**（ROADMAP 1.2） |
| DQC 监控 loop | 人按回手 | panel_data_quality ✅ | 表 ✅ | — | **半自动**，缺定时触发 |

> **结论**：我们的手艺（探查→mock→回补→EDA）结构上已经是 loop，只是**触发器是人**、**验证门散落在脑子里没规则化**。这次改造做的就是把验证门收口成硬规则（`.trae/rules/nanoquant_loop.md`），让 AI 每轮都过门，而不是靠作者每次口头叮嘱。

## 8. 以后怎么扩（给作者的建议路径）

1. **先把现有隐性 loop 的验证门钉死**（本次已做 → 项目规则）。AI 每轮自动过门，不用你盯。
2. **策略层验证门是最大缺口**（ROADMAP 1.1 真实回测 + 1.2 因子检验）。这两个是"能说不的东西"，建好前信号不可信。
3. **DQC loop 加触发器**：panel_data_quality 表已有，配个 Windows 任务计划定时跑，就是第一个真·自动 loop。但**先手动跑稳了再上定时**（配方第 5 条）。
4. **暂不上全自动调度**（prefect 之类）。个人单兵 + 不定期开机，人按回车的节奏够了。等某个手动 loop 跑了几十轮都稳，再考虑自动化它。
5. **每个新 loop 落地前，过一遍 §5 的 5 条自检**，写进项目规则 §7 那一节。
